Пользователи и права доступа

Обзор системы пользователей

EmDash имеет встроенную систему аутентификации и авторизации:

  • Аутентификация (кто вы) — локальная (email+пароль) или OAuth (Google, GitHub)
  • Авторизация (что можно) — роли и permissions
  • Сессии — управляются Astro (@astrojs/node)

Хранение: данные пользователей в таблице users (SQLite).
Хеширование паролей: Argon2 или bcrypt (EmDash default).

Роли пользователей

Роль 英文 Права
Admin Администратор Полный доступ: создание/удаление контента, управление пользователями, настройки сайта, плагины
Editor Редактор Создание/редактирование любого контента (pages, posts, teachers...), но не настройки и не пользователи
Author Автор Создание/редактирование своих записей только. Не может публиковать чужие.
Viewer Зритель Только чтение контента, без прав на изменение

По умолчанию: при первом запуске создаётся пользователь admin@rksh-msk.ru с ролью Admin.

Создание пользователя

Через админку

  1. /_emdash/admin → Пользователи (Users)
  2. «+ Добавить пользователя»
  3. Заполнить:
    • Имя (display name)
    • Email (логин)
    • Пароль (минимум 8 символов)
    • Роль (Admin/Editor/Author/Viewer)
  4. «Сохранить»

Через CLI (emdash)

npx emdash user:create --email="editor@rksh.ru" --role=editor --name="Иван Редактор"

Аутентификация

Локальная (email + password)

Стандартная форма входа /login.
EmDash хранит хеш пароля,.compare на каждом запросе.

Требования к паролю:

  • Минимум 8 символов
  • Хотя бы одна заглавная буква
  • Хотя бы одна цифра
  • (опционально) special character

OAuth (опционально)

Настройка в astro.config.mjs:

emdash({
  auth: {
    providers: [
      {
        type: "google",
        clientId: process.env.GOOGLE_CLIENT_ID,
        clientSecret: process.env.GOOGLE_CLIENT_SECRET,
      },
      {
        type: "github",
        clientId: process.env.GITHUB_CLIENT_ID,
        clientSecret: process.env.GITHUB_CLIENT_SECRET,
      },
    ],
  },
});

Плюсы: пользователи не помнят пароли, быстрая регистрация.
Минусы: зависимость от сторонних сервисов.

Сессии

Механизм: @astrojs/node → файловая система (dev) или Redis (prod).

Конфигурация:

// astro.config.mjs
adapter: node({
  mode: "standalone",
  session: {
    // В production использовать Redis
    // driver: "redis",
    // redis: { url: process.env.REDIS_URL }
  },
}),

TTL сессии: 7 дней (по умолчанию). Можно настроить.

Управление пользователями

Список пользователей

Админка → Пользователи. Таблица:

  • Email
  • Имя
  • Роль
  • Статус (active / disabled)
  • Дата последнего входа

Изменение роли

  1. Нажать на пользователя
  2. Выбрать новую роль из dropdown
  3. Сохранить

Блокировка пользователя

  • Disable — вход запрещён, но записи сохраняются
  • Delete — полное удаление (каскадное? нужно проверить)

Аудит и логи

Логи действий: EmDash может вести audit log (через плагин audit-log или свой).

Что логировать:

  • Вход/выход
  • Создание/редактирование/удаление записей
  • Изменение настроек
  • Загрузка/удаление медиа

Хранение: отдельная коллекция audit_log или внешний сервис (Sentry).

Многофакторная аутентификация (MFA)

Пока не реализована. Можно добавить:

  • TOTP (Google Authenticator)
  • SMS-коды
  • Email-ссылки

Настройка: в админке → настройки безопасности.

Сброс пароля

Механизм:

  1. Пользователь нажимает «Забыли пароль?»
  2. Вводит email
  3. EmDash отправляет ссылку с токеном (срок действия 1 час)
  4. Переход по ссылке → форма нового пароля

Настройка SMTP обязательна для отправки писем.

Настройки безопасности

В .env

# Секрет для сессий (должен быть надёжным)
EMDASH_SECRET=supersecurekeychangeit

# Сложность пароля (опционально)
PASSWORD_MIN_LENGTH=8
PASSWORD_REQUIRE_UPPER=true
PASSWORD_REQUIRE_NUMBER=true

# Rate limiting
RATE_LIMIT_MAX=100  # запросов в час с IP

В коде

// Максимальные попытки входа
const MAX_LOGIN_ATTEMPTS = 5;
// Блокировка на 15 минут после 5 неудачных

IP-фильтрация (опционально)

Ограничить доступ к админке только с определённых IP:

// src/middleware.ts
export const onRequest = async (context) => {
  const { url } = context;
  if (url.pathname.startsWith("/_emdash/admin")) {
    const ip = context.request.headers.get("x-forwarded-for") || "127.0.0.1";
    const allowed = process.env.ADMIN_IPS?.split(",") || ["127.0.0.1"];
    if (!allowed.includes(ip)) {
      return new Response("Forbidden", { status: 403 });
    }
  }
};

Резервное копирование

Что бэкапить

  1. База данных (data.db) — весь контент, пользователи, настройки
  2. Папка uploads/ — медиафайлы
  3. seed.json — схема (можно восстановить структуру)

Стратегия

Ежедневный бэкап:

# Скрипт backup.bat
sqlite3 data.db ".backup backup/data-$(date +%Y%m%d).db"
xcopy uploads backup\uploads-$(date +%Y%m%d) /E /I

Хранение: облако (Yandex Cloud, S3) + локально.

Восстановление

  1. Остановить сервер
  2. Заменить data.db на бэкап
  3. Восстановить uploads/
  4. Запустить npm start

Аудит изменений

Включение аудита

Создать/включить плагин audit-log (если не вшит):

import { auditLogPlugin } from "emdash/plugins/audit-log";
emdash({
  plugins: [auditLogPlugin()],
});

Просмотр логов

Админка → Аудит (или запрос к API):

const { entries } = await getEmDashCollection("audit_log", {
  limit: 100,
  orderBy: { timestamp: "desc" },
});

Частые вопросы

Как изменить пароль admin?

  1. Войти под admin
  2. Профиль → Сменить пароль

Если забыли пароль — через CLI:

npx emdash user:reset-password --email=admin@rksh.ru

Как удалить пользователя?

Важно: нельзя удалить последнего Admin. Сначала назначить другого.

  1. Админка → Пользователи
  2. Выбрать пользователя → Удалить
  3. Подтвердить

Эффект:

  • Записи автора (posts, pages) сохраняются, но поле created_by станет null
  • Не каскадное удаление контента

Best practices

  1. Никому не давать Admin — только владельцу/техническому специалисту
  2. Editor — для контент-менеджеров (редактируют всё, но не настройки)
  3. Author — для внешних сотрудников (только свои материалы)
  4. Отключить неактивных через 90 дней неиспользования
  5. Обязательные логи: вход, выход, изменение пароля
  6. Регулярный аудит (раз в квартал) — кто что менял

Интеграция с Active Directory / LDAP

Пока не поддерживается. Можно написать кастомный auth provider:

// plugins/ldap-auth/src/index.ts
export function ldapAuth(options) {
  return definePlugin({
    auth: {
      providers: [
        {
          type: "custom",
          name: "LDAP",
          handler: async (credentials) => {
            // проверить в LDAP
            // вернуть user object
          },
        },
      ],
    },
  });
}

  • Логгирование в audit_log

Усиление:

  • reCAPTCHA на форме входа
  • Уведомление email при новом входе
