Контексты выполнения кода

Build-time (Vite)

  • astro.config.mjs — импорты, конфигурация
  • PluginDescriptor (из plugins/*/src/index.ts) — описание плагина без side-effects
  • Запускается при сборке/старте dev-сервера

Request-time (SSR)

  • definePlugin() (из plugins/*/src/sandbox-entry.ts) — логика плагина
  • EmDash middleware — обработка запросов к /_emdash/api/*
  • Astro page components — рендеринг страниц
  • Запускается на каждый HTTP-запрос (в продакшн)

Admin UI (браузер)

  • React-приложение на /_emdash/admin
  • Подключается к API EmDash
  • Требует EMDASH_SECRET для аутентификации

Поток данных

1. Запрос страницы → Astro middleware
2. EmDash проверяет кэш (cacheHint)
3. Если кэш невалиден → запрос к БД (SQLite)
4. Контент форматируется → PortableText → block components (если есть кастомные блоки)
5. Астро-компоненты рендерятся в HTML
6. Ответ отправляется браузеру

Ключевые принципы

  1. Server-side rendering: все страницы с контентом — SSR, нет статической генерации для CMS-страниц
  2. Кэширование: каждый запрос контента возвращает cacheHint → Astro.cache.set(cacheHint) обязателен
  3. Изоляция контекстов: descriptor (build) ≠ definition (runtime)
  4. Плагины как пакеты: @rksh/blocks — workspace package, подключается через plugins: в astro.config.mjs

Жизненный цикл запроса (подробно)

1. Пользователь → http://localhost:4321/o-shkole
2. Astro router → находит src/pages/pages/[slug].astro
3. getEmDashEntry("pages", "o-shkole") → запрос к EmDash API
4. EmDash → проверяет кэш (Redis/in-memory)
5. Если miss → query SQLite: SELECT * FROM entries WHERE collection='pages' AND slug='o-shkole'
6. Полученная запись → преобразуется в entry object с .data
7. Astro компонент рендерит `<Base> <PortableText value={entry.data.content} />`
8. PortableText проходит по массиву блоков:
   - Если `_type === "rksh.hero"` → импортирует `Hero` из `@rksh/blocks/astro`
   - Передаёт `node` (все поля) в компонент
9. Компонент возвращает HTML → собирается вся страница
10. Ответ кэшируется с `cacheHint` (TTL из настроек коллекции)
11. Следующий запрос → кэш hit → молниеносно

Типы данных EmDash

// Entry (запись)
{
  id: string;          // slug (для URL)
  data: {
    id: string;        // ULID (для API)
    title: string;
    content: PortableText[];
    publishedAt: Date | null;
    // ... custom fields
  };
  status: "draft" | "published" | "archived";
  version: number;
  publishedAt: Date | null;
  cacheHint: { ttl: number; tags: string[] };
}

// Menu item
{
  label: string;
  url: string;
  target?: string;
  children: MenuItem[];
}

// Settings (site-wide)
{
  site_name: string;
  logo: { src: string; alt: string };
  phone_primary: string;
  ...
}

Ограничения и gotchas

  1. Images are objects: { src, alt }, не строки. Использовать <Image image={...} /> из emdash/ui
  2. entry.id ≠ entry.data.id: первый — slug, второй — ULID
  3. Taxonomy name exact match: если в seed "name": "category", то getTerm("category", slug), а не "categories"
  4. Cache hint mandatory: без Astro.cache.set(cacheHint) контент не инвалидируется при публикации
  5. No getStaticPaths(): CMS-контент не может быть статически сгенерирован
